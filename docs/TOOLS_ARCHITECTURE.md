# Omni One Tool Architecture (Phase 12)

This document details the production-ready architecture of the **Universal Tool SDK**, **Native Tool Library**, and **Tool Runtime System** implemented in Phase 12.

## 1. Universal Tool SDK (Phase 12.1)

The Universal Tool SDK defines the core contract for all tools in the Omni One system. It is designed to be fully decoupled from the core runtime, allowing any future tool to be added dynamically.

### Core Interface (`IToolSDK`)
Every tool MUST implement `IToolSDK`, which exposes:
- **`initialize()`**: Warm up connections and load models.
- **`validate()`**: Validate inputs against the JSON schema.
- **`execute()`**: Run the tool and return a standardized `ToolExecutionResult`.
- **`stream()`**: Stream outputs chunk by chunk.
- **`cancel()`**: Abort an ongoing execution.
- **`cleanup()`**: Free resources on shutdown.
- **`healthCheck()`**: Verify tool availability.

### Base Class (`AbstractToolSDK`)
To eliminate boilerplate, tools extend `AbstractToolSDK`. This base class automatically handles execution lifecycle, timeout injection via `AbortController`, error catching, and state transitions (`idle` → `executing` → `ready`).

## 2. Native Tool Library (Phase 12.2)

The system includes a comprehensive suite of native tools, all built on top of the Universal Tool SDK.

### Available Tool Categories:
1. **Browser Tools**: Navigate, Read Page, Extract Text, Extract Tables, Extract Links, Screenshot.
2. **File & Image Tools**: Read PDF, DOCX, TXT, CSV, Excel, Markdown, JSON. Image OCR, Metadata, Resize, Crop, Format Conversion.
3. **Audio & Video Tools**: Speech-to-Text (Whisper), Text-to-Speech, Video Metadata, Frame Extraction.
4. **Collaboration Tools**: GitHub (Read Repo, Read File, Create Commit, Create PR, Issues), Email (Send, Read, Search), Calendar (Create, Update, Delete).
5. **HTTP & Database Tools**: REST, GraphQL, Webhooks. Unified Database query tool (SQLite, PostgreSQL, MySQL).

## 3. Capability Registry Integration (Phase 12.3)

Manual registration is completely eliminated.
- Tools automatically register themselves into `ToolSDKRegistry` upon module import.
- `CapabilityRegistryIntegration` bridges the `ToolSDKRegistry` with the legacy `CapabilityRegistry`.
- The AI uses the `ToolDiscovery` API to dynamically search for tools based on natural language intent, tags, or required permissions.

## 4. Permission System (Phase 12.4)

Security is enforced at runtime before any tool executes.
- **Declaration**: Every tool statically declares required permissions (e.g., `read`, `write`, `network`, `filesystem`, `database`) and explicitly flags `dangerousPermissions`.
- **Validation**: The `PermissionValidator` checks the active execution context against the tool's requirements.
- **Runtime Guard**: `RuntimePermissionGuard` wraps execution. If a tool requests a dangerous permission (like `write` or `execute`), the system requires explicit user confirmation before proceeding.

## 5. Tool Planner (Phase 12.5)

Before executing a complex task, the AI generates an `ExecutionPlan`.
- **Strategy Selection**: The planner analyzes dependencies between steps to choose between `sequential`, `parallel`, or `pipeline` execution.
- **Resilience**: Each step is assigned a `RetryPolicy` (max attempts, backoff).
- **Rollback**: Tools can define `RollbackAction`s. If a critical step fails, the `ToolExecutor` automatically executes rollbacks for previously completed steps in reverse order.

## 6. Tool Result Fusion (Phase 12.6)

When multiple tools return data for the same goal, the `ToolResultFusion` engine:
1. **Merges**: Combines structured data and text outputs.
2. **Deduplicates**: Hashes data payloads to remove overlapping results from different tools.
3. **Ranks**: Boosts confidence scores for items corroborated by multiple sources.
4. **Validates**: Ensures the final fused output meets quality thresholds before returning it to the user.

## 7. Tool Marketplace Architecture (Phase 12.7)

The `ToolMarketplace` handles the lifecycle of external tool packages.
- **Installation**: Resolves dependencies, checks SDK version compatibility, and installs packages.
- **Versioning**: Supports semantic versioning and safe updates (install new → remove old).
- **Sandboxing**: Sandboxed tools are assigned isolated execution environments (`ToolSandboxConfig`) with strict memory, CPU, and network limits.
